# Улучшение ранжирования проектов под требования вакансии

## Проблема

При ранжировании проектов под вакансию «Backend Developer (FastAPI, Nuxt3, PostgreSQL, Docker & Kubernetes, NATS, Redis, WebSockets, Celery)» проект **Midas trading**, содержащий явный стек `Python + FastAPI + Docker`, оказался на 4-м месте (score 0.7308), а топ занял **LOKALI APP** на `Node.js + MySQL + RabbitMQ` (score 0.7397) — без FastAPI вообще.

### Пример выдачи

| # | Проект | Score | Релевантный стек |
|---|--------|-------|------------------|
| 1 | LOKALI APP | 0.7397 | Node.js, MySQL, RabbitMQ |
| 2 | GEFEST Capital | 0.7346 | PHP, Node.js, PostgreSQL |
| 3 | THEDED IT | 0.7342 | PHP, Node.js, Python (Django, FastAPI) |
| 4 | **Midas trading** | **0.7308** | **Python, Django, FastAPI, Docker** |
| 5 | Remote work | 0.7190 | Docker, Linux |

## Причины

Ранжирование сейчас построено на **чистой косинусной близости плотных эмбеддингов** (dense vector similarity). Системные проблемы:

### 1. Размер и насыщенность текста проекта
- **LOKALI APP** — 5 развёрнутых достижений, многократное упоминание `REST API`, `RESTful API`, `mobile applications`, `S3`, `geospatial`. Эмбеддинг семантически богатый.
- **Midas trading** — 1 короткое достижение, технологии перечислены строкой.

Эмбеддинг короткого текста размывается и проигрывает длинному, даже если в длинном **нет ключевой технологии**.

### 2. Эмбеддинг не понимает, что `FastAPI` — критический ключ
Для модели `FastAPI`, `Django`, `Flask`, `REST API` — семантически близкие сущности (все про Python web). Проект на **Node.js + REST API** получает близкий скор к Python + FastAPI. Косинусная близость не различает «обязательный стек» и «общая тема».

### 3. Неточная нормализация технологий
В payload встречаются разные формы:
- `"fast api"` (Midas)
- `"python (django, fast api)"` (THEDED — целая строка как один токен)
- Требование вакансии: `FastAPI`

Это разные токены. Никакой exact-match boost не сработал бы.

### 4. Смешение языков
Требования на английском, описания проектов на русском (местами с битой кодировкой: `бизnesс-часов`, `геоспоциальное`). Многоязычная модель работает, но качество просаживается на коротких текстах.

### 5. Нет учёта актуальности / даты проекта
Свежесть проекта не влияет на ранжирование.

## Рекомендации по архитектуре поиска

### A. Гибридный поиск (must-have)
Qdrant поддерживает named vectors + sparse. Использовать **dense + sparse (BM25/SPLADE)** и сливать через **RRF (Reciprocal Rank Fusion)**:
- Dense ловит семантику (`backend developer` ≈ `разработчик бэкенда`).
- Sparse ловит точные ключи: `FastAPI`, `Celery`, `NATS`, `Kubernetes`.

### B. Жёсткое извлечение требований и матчинг по тегам
Перед ранжированием (LLM-ом или регулярками) вытащить из вакансии **canonical tech list**:
```json
["fastapi", "postgresql", "redis", "celery", "kubernetes", "websockets", "nats"]
```

Для каждого проекта считать **tech_overlap_score**:
```
overlap = |required ∩ project.tech_normalized| / |required|
```

Финальный скор:
```
final = α * cosine + β * overlap + γ * recency
```
α≈0.4, β≈0.5, γ≈0.1 (подбирается эмпирически).

### C. Нормализация технологий (фикс существующего поля)
В `tech_normalized` привести к единому канону:
- `"fast api"`, `"fastapi"`, `"FastAPI"` → `fastapi`
- `"python (django, fast api)"` — **разбивать на токены**, а не хранить как одну строку.
- Убирать версии: `"java 17"` → `"java"`, `"php 8+"` → `"php"`.

Без этого никакой keyword-match не сработает.

### D. Reranker (cross-encoder) поверх top-K
После гибридного поиска брать top-20 и прогонять через cross-encoder (например, `bge-reranker-v2-m3` — многоязычный). Он сравнивает пары `(требования вакансии, текст проекта)` целиком и даёт более точный скор, чем cosine.

### E. Поле-зависимое взвешивание
Сейчас весь проект эмбеддится в один текст. Лучше:
- отдельный вектор для `technologies`
- отдельный для `achievements`
- финальный скор = взвешенная сумма (технологии × 0.6, достижения × 0.4)

Это решит проблему «короткое описание → размытый эмбеддинг».

### F. Query expansion
Перед поиском расширять требования синонимами через LLM:
```
FastAPI → [FastAPI, ASGI, Starlette, async Python web, Uvicorn]
```
Эмбеддинг расширенного запроса лучше «цепляет» релевантные проекты.

### G. Hard filter, а не только скор
Если в вакансии явно сказано «3 года FastAPI» — добавить **hard filter** в Qdrant:
```
must: tech_normalized contains "fastapi"
```
Тогда Node.js-проекты не попадают в выдачу, ранжируется только релевантный пул.

## Минимальный практический рецепт

1. Починить `tech_normalized` (токенизация + канонизация словаря).
2. Извлекать `required_tech` из вакансии (LLM-парсер требований уже есть — добавить поле `tech_keys`).
3. Использовать Qdrant **с фильтром** `should` по `tech_normalized` + dense поиск.
4. Поверх — пересчёт скора с бонусом за overlap технологий.
5. На втором этапе — cross-encoder reranker по top-10.

После шагов 1–4 проект с FastAPI поднимется в топ; шаг 5 даст финальное качество.
