# Ускорение генерации Cover Letter — исследование и план

## Текущий пайплайн (как работает сейчас)

```
URL вакансии
  │
  ▼
[1] gpt-4.1-mini + web_search_preview → парсинг требований (~3-8 сек)
  │
  ▼
[2] embed_texts("ключевые навыки опыт образование достижения") → OpenAI embedding API (~0.5-1 сек)
  │
  ▼
[3] Qdrant search top_k=10 → фильтрация по source_id в Python (~0.05 сек)
  │
  ▼
[4] Сборка промпта (requirements + resume_context) → gpt-4o → генерация письма (~5-15 сек)

Итого: ~10-25 секунд до полного ответа (без стриминга)
```

### Выявленные узкие места

| # | Проблема | Файл | Влияние |
|---|----------|------|---------|
| 1 | Два последовательных LLM-вызова (парсинг URL → генерация) | `services/letter.py` L162-180, L58-110 | x2 латентность |
| 2 | Статический запрос для векторного поиска `"ключевые навыки опыт образование достижения"` — не зависит от вакансии | `services/letter.py` L82 | Плохая релевантность чанков → LLM тратит токены на нерелевантный контекст |
| 3 | Фильтрация по `source_id` на стороне Python, а не Qdrant | `services/letter.py` L75-80 | Если 7 из 10 результатов чужие — остаётся 3 чанка вместо 10 |
| 4 | `gpt-4o` для генерации 200-300 слов — избыточно | `services/letter.py` L105 | Медленнее и дороже чем нужно |
| 5 | Синхронный вызов embeddings в async контексте | `services/pdf.py` L55-61 | Блокирует event loop |
| 6 | Нет кэширования (URL парсинг, embeddings, поиск) | — | Повторные запросы к тому же URL = те же 10-25 сек |
| 7 | Embedding dim=3072 — избыточно для поиска по резюме | `services/pdf.py` L13-14 | Медленнее поиск в Qdrant |

## Предложения по ускорению

### 🔴 Высокий приоритет (максимальное влияние на скорость)

#### 1. Qdrant server-side фильтрация по `source_id`

**Проблема**: Сейчас ищем top_k=10 по всей коллекции, затем отбрасываем чужие записи в Python. Если в БД 100 пользователей — 9 из 10 результатов могут быть чужими.

**Решение**: Передавать `source_id` фильтр прямо в Qdrant `query_points()`:

```python
# storage/repository/qdrant.py — добавить параметр filter
from qdrant_client.models import Filter, FieldCondition, MatchValue

def search(self, query_vector, top_k=5, source_id=None):
    search_filter = None
    if source_id is not None:
        search_filter = Filter(
            must=[FieldCondition(key="source_id", match=MatchValue(value=source_id))]
        )
    results = self.client.query_points(
        collection_name=self.collection,
        query=query_vector,
        query_filter=search_filter,
        with_payload=True,
        limit=top_k
    ).points
    ...
```

**Эффект**: Всегда получаем ровно top_k релевантных чанков пользователя. Убирается Python-фильтрация.

**Сложность**: Низкая. Изменения в 2 файлах.

---

#### 2. Использовать требования вакансии как запрос для векторного поиска

**Проблема**: Статический запрос `"ключевые навыки опыт образование достижения"` для embed → search. Для вакансии Python Developer и для вакансии Marketing Manager возвращаются одни и те же чанки из резюме.

**Решение**: Использовать ключевые слова/навыки из `job_requirements` как запрос в Qdrant:

```python
# Вместо:
skills_query = "ключевые навыки опыт образование достижения"

# Использовать:
skills_query = job_requirements[:500]  # или извлечь ключевые слова
```

**Эффект**: Резюме-контекст становится релевантным вакансии → LLM генерирует быстрее и точнее (меньше шума в промпте).

**Сложность**: Минимальная. 1 строка кода.

---

#### 3. Переход на быструю модель для генерации письма

**Проблема**: `gpt-4o` для генерации 200-300 слов — как стрелять из пушки по воробьям. Основное качество зависит от контекста (хорошие чанки из резюме), а не от reasoning-способностей модели.

**Решение**: Заменить `gpt-4o` на `gpt-4.1-mini` или `gpt-4.1-nano` для генерации.

```python
# services/letter.py
response = self.client.responses.create(
    model="gpt-4.1-mini",  # было gpt-4o
    ...
)
```

**Эффект**: ~2-4x быстрее, ~10x дешевле. Качество при хорошем контексте остаётся высоким.

**Сложность**: Минимальная. Смена 1 строки + тестирование качества.

#### 4. Кэширование результатов парсинга URL вакансии

**Проблема**: Каждый запрос к тому же URL = новый вызов `gpt-4.1-mini` + web_search. Парсинг URL — самая медленная фаза (~3-8 сек).

**Решение**: Redis/in-memory кэш с TTL (например, 1 час) по URL:

```python
from functools import lru_cache
import hashlib

# Простой вариант — in-memory dict с TTL
_url_cache: dict[str, tuple[str, float]] = {}
CACHE_TTL = 3600  # 1 час

async def _parse_job_requirements_from_url(self, job_url: str) -> str:
    cache_key = hashlib.sha256(job_url.encode()).hexdigest()
    if cache_key in _url_cache:
        cached, ts = _url_cache[cache_key]
        if time.time() - ts < CACHE_TTL:
            return cached
    result = ...  # LLM вызов
    _url_cache[cache_key] = (result, time.time())
    return result
```

**Эффект**: Повторный запрос к тому же URL = мгновенный ответ (~0 мс вместо 3-8 сек).

**Сложность**: Низкая.

### 🟡 Средний приоритет

#### 5. Предварительное извлечение структурированных данных из резюме (при загрузке CV)

**Проблема**: При каждой генерации — embed query → Qdrant search → собрать текстовые чанки. Структура данных теряется.

**Решение**: При загрузке PDF дополнительно извлекать структурированные данные (навыки, проекты, опыт) и хранить в PostgreSQL:

```python
# При upload CV:
# 1. Стандартный pipeline: PDF → chunks → Qdrant (для семантического поиска)
# 2. Новый шаг: отправить полный текст в LLM → JSON со структурой:
structured = {
    "skills": ["Python", "FastAPI", "React", ...],
    "projects": [
        {"name": "RAG System", "tech": ["Python", "Qdrant"], "description": "..."},
    ],
    "experience": [
        {"company": "...", "role": "...", "period": "...", "highlights": ["..."]},
    ]
}
# Сохранить в PostgreSQL (новое поле в CV или отдельная таблица)
```

**Эффект**: При генерации письма можно пропустить embed+search и сразу использовать структурированные данные. Генерация из pre-computed данных = ~0 мс на этап RAG.

**Сложность**: Средняя. Новая модель/миграция + логика извлечения.

#### 6. Хранение проектов с ключевыми словами в Qdrant (предложение пользователя)

**Проблема**: Сейчас чанки в Qdrant — это сырые куски текста из PDF. Нет структуры «проект — навыки — описание».

**Решение**: При загрузке CV парсить проекты/опыт в отдельные структурированные записи и хранить в Qdrant с enriched payload:

```python
# Новая структура payload в Qdrant
{
    "user_id": 1,
    "source_id": "cv_123",
    "type": "project",          # project | skill | experience
    "text": "Разработал RAG-систему на FastAPI + Qdrant для автоматизации...",
    "keywords": ["Python", "FastAPI", "Qdrant", "RAG", "OpenAI"],
    "project_name": "RAG Cover Letter Generator",
    "technologies": ["Python", "FastAPI", "Qdrant"],
    "chunk_index": 0
}
```

При генерации — LLM получает готовые структурированные блоки, которые просто нужно отсортировать по релевантности и вставить в письмо.

**Эффект**: LLM тратит меньше токенов на «понимание» резюме → быстрее генерация. Качество выше за счёт структурированности.

**Сложность**: Средняя. Новый pipeline обработки PDF + миграция payload-структуры.

#### 7. Уменьшение промпта + акцент на данных из БД (предложение пользователя)

**Проблема**: Промпт содержит и инструкции, и контекст, и требования — всё в одном блоке ~500-1000 токенов на вход.

**Решение**: 
- Системный промпт — минимальный и фиксированный (роль + формат)
- Данные пользователя — структурированно из БД (skills, projects)
- User message — только требования вакансии

```python
system_prompt = """Ты создаёшь сопроводительные письма. 
Формат: 200-300 слов, профессиональный тон, на языке вакансии.
Сопоставь опыт кандидата с требованиями."""

user_prompt = f"""
Требования вакансии:
{job_requirements}

Навыки кандидата: {skills_from_db}
Проекты: {projects_from_db}
Опыт: {experience_from_db}
"""
```

**Эффект**: Меньше input-токенов → быстрее отклик. LLM не тратит время на разбор сырого текста.

**Сложность**: Низкая (промпт) + средняя (структурированные данные из БД).

#### 8. Параллельный запуск embedding и парсинга URL

**Проблема**: Шаги идут строго последовательно: парсинг URL → embedding → search → генерация.

**Решение**: В `stream_by_url` запускать парсинг URL и embedding запроса параллельно через `asyncio.gather`:

```python
import asyncio

# Парсим URL и готовим embedding параллельно (для text flow)
async def generate_cover_letter(self, job_requirements, source_id):
    # Запускаем embedding в отдельном потоке (он синхронный)
    loop = asyncio.get_event_loop()
    embed_task = loop.run_in_executor(
        None, self.pdf_service.embed_texts, [job_requirements[:500]]
    )
    # ... остальная логика
    query_vec = await embed_task
```

**Эффект**: Экономия ~0.5-1 сек на embedding.

**Сложность**: Низкая.

### 🟢 Дополнительные оптимизации

#### 9. Уменьшение размерности embeddings

**Текущее**: `text-embedding-3-large` с dim=3072. Для поиска по личному резюме (десятки чанков, не миллионы) — избыточно.

**Решение**: Уменьшить до 1024 или 512 (API поддерживает параметр `dimensions`):

```python
EMBED_DIM = 1024  # было 3072
```

**Внимание**: Требует пересоздания коллекции Qdrant и ре-эмбеддинга всех CV.

**Эффект**: ~2-3x быстрее поиск в Qdrant, меньше места, embedding API быстрее.

**Сложность**: Средняя (миграция данных).

---

#### 10. Кэширование embedding для статического запроса

**Проблема**: Если запрос для поиска статический (`"ключевые навыки опыт образование достижения"`) — его embedding вычисляется заново при каждом запросе.

**Решение**: Закэшировать embedding для повторяющихся запросов. (Актуально, пока не реализован п.2 — динамический запрос.)

**Эффект**: -0.5-1 сек на запрос.

**Сложность**: Минимальная.

---

#### 11. Использовать `asyncio.to_thread` для синхронных вызовов OpenAI

**Проблема**: `self.pdf_service.embed_texts()` — синхронный вызов OpenAI SDK, блокирует event loop.

**Решение**:

```python
import asyncio

query_vec = await asyncio.to_thread(self.pdf_service.embed_texts, [query])
```

Или перейти на `AsyncOpenAI` клиент для embeddings.

**Эффект**: Не блокирует другие запросы на сервере (улучшает throughput, не latency одного запроса).

**Сложность**: Минимальная.
